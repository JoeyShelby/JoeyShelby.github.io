1 项目简介

这是一个十分简单的基于 SSM 框架的 CRUD 项目，目的在于学习完 Spring, SpringMVC, Mybatis 后熟练三者的整合使用。

Github地址：JoeyShelby/SSM-CRUD

3 数据表设计

该项目用于管理员工信息，设计如下数据表：

t_emp : 员工表

t_dept : 部门表

其中 t_emp 中的 dept_id 依赖于 t_dept 中的 dept_id， 多对一关系。

注意在设计数据表时，使用下划线命名法，以配合 Mybatis 将其转换成驼峰命名规则。

4 逆向工程

Mybatis 提供了一个逆向工程工具，该工具可以根据数据表自动生成针对单表的 po 类、mapper 映射文件和 mapper 接口。大大缩减了开发时间，可以让开发人员将更多的精力放在繁杂的业务逻辑上。

4.2 修改

使用 MGB 自动生成的 po 类、mapper 映射文件和 mapper 接口 往往并不会完全满足要求，需要根据需求对这些自动生成的文件进行修改。

比如在本项目中，要求查询员工信息时，同时查询出对应的部门信息，那么就需要修改自动生成的 po 类、mapper 映射文件和 mapper 接口。

6 service 层设计

实际上，service 层的设计常常并不是一步到位的，而是根据需求，再添加 service 层中的业务逻辑。

6.1 EmployeeService

  1. 查询所有员工信息
  2. 添加员工信息
  3. 检验员工名是否可用（员工名唯一）
  4. 按 ID 查询员工信息
  5. 员工信息更新
  6. 删除员工信息

6.2 DepartmentService

  1. 查询所有部门信息

7 service 层测试

service 层的测试同样是十分重要的，确保 service 层的业务可以正常执行，为 contoller 层打好基础。

这里的测试使用 Spring5 测试配合 Junit 单元测试。

8 前端页面设计

前端页面采用 BootStrap 4 前端框架配合 Jquery 库进行开发设计。

前后端数据交互使用 JSON 格式，使用 JSON ，可以大大降低前后端之间的耦合度，前后端之间只进行数据之间的交互，这样也能够适应于不同的前端，比如 Android 等，无需修改后端接口。

9 功能实现

R - 查询员工信息及分页

后端

后端使用 PageHelper 插件，方便地实现数据的分页查询以及导航条等分页信息。最终这些信息都封装到一个 PageInfo 对象中，用 JSON 的方式发送给前端。

前端

使用 ajax 请求从后端获取到分页信息之后，再将员工信息以及分页信息动态地加载到员工信息表格以及分页信息容器中。

C - 新增员工信息

后端

后端获取从前端传入的员工信息对象，再次进行数据校验之后，向数据库中插入新纪录。这里用了 JSR303 校验。

前端

  1. 使用 BootStrap 4 中的模态框，设计如上的添加功能模态框。

  2. 打开模态框之后，对【部门】下拉菜单进行填充。具体实现是通过发送 ajax 请求，获取到全部的部门信息，再动态地填充到【部门下来菜单中】。

  1. 对输入的信息进行校验，如 姓名 要满足姓名规则，邮箱地址必须满足规则，必须要选择部门等。只有数据验证通过之后，才可以向后端提交添加新员工的请求。
  1. 添加成功后，关闭模态框，提示添加成功，并使分页跳转到最后一页。

U - 更新员工信息

后端

总体逻辑与新增员工信息类似，需要先验证要修改的信息是否合法，然后根据员工 id ，更改其对应信息。

前端

  1. 给每一条员工信息的 【修改】按钮绑定单击事件，以弹出模态框。这里有一个很重要的细节：在生成各条员工信息时，为【修改】按钮添加两样重要的东西。一是为其添加一个自定义 class，我这里取名为”edit_btn“，以方便之后为每一个【修改】按钮绑定弹出模态框的单击事件；二是添加一个自定义属性 emp-id，每一条员工信息的【修改】按钮都带有对应的员工 id ，这样就能很方便地做到用户信息的回显。
  2. 回显用户信息。
  3. 对修改的信息进行校验。
  4. 通过 ajax 发送更新请求。

D - 删除员工信息

后端

前端
