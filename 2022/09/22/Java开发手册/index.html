Java 开发手册笔记

发表于 2022-09-22 分类于 学习 ， Java Valine：

编程规约

命名风格

  1. 不使用 _ 或 & 开始或结束。

  2. 避免拼音。

  3. 常量命名全部大写，使用 _ 分隔开。

    如：MAX_STOCK_COUNT

  4. 类名规约：

    1. 采用 UpperCamelCase。
    2. 抽象类以 Abstract 或 Base 开头。
    3. 异常类命名使用 Exception 结尾。
    4. 测试类以被测试的类的名称开始，以 Test 结尾。

  5. 数组定义应使用如下形式，而非 C 形式。

    String[] args;

  6. POJO 类中布尔类型变量不加 is。

  7. 避免缩写，尽量使用完整词组。

  8. 接口中方法和属性不加任何修饰，尽量避免在接口中定义变量。

  9. 枚举类：

    类名加上 Enum 后缀

    枚举成员全大写，单词间下划线隔开

  10. Service / DAO 层方法名：

    获取单个对象：get

    获取多个对象：list

    获取统计值：count

    插入：save

    删除：remove

    修改：update

常量定义

  1. 不允许使用魔法值。

  2. long 或 Long 初始赋值时，必须使用大写L，避免l与1混淆。

    Long a = 2L;

  3. 常量类应当使用不同常量类进行维护，而不是只是用一个常量类维护所有常量。

    缓存相关常量放在类 CacheConsts 下；系统配置相关常量放在类 ConfigConsts 下。

  4. 如果一个变量仅在一个小范围内变化，则应当定义为枚举类。

    正例：public Enum { MONDAY(1), TUESDAY(2), WEDNESDAY(3), THURSDAY(4), FRIDAY(5), SATURDAY(6), SUNDAY(7);}

代码格式

  1. 大括号：

    1. 空大括号不换行，{}

    2. 非空代码块

      • 左大括号前不换行
      • 右大括号前换行
      • 右大括号后还有 else 等代码则后不换行
      • 右大括号表示终止则必须换行

  2. 小括号：

    if/for/while/switch/do 等保留字与括号之间都必须加空格。

  3. 任何二目、三目运算符的左右两边都需要加一个空格。

    if (flag == 0)

  4. 缩进采用 4 个空格，禁止使用 tab 字符。

  5. 注释的双斜线与注释内容之间有且仅有一个空格。

  6. 单行字符过长需要换行。

    1. 第二行缩进 4 个空格，第三行开始不再缩进。
      2. 运算符与下文一起换行。
      3. 方法调用点号与下文一起换行。
      4. 方法传参时，多个参数换行，在逗号后进行。
      5. 括号前不要换行。

  7. 方法传参时，参数逗号后面必须加空格。

  8. 不同业务逻辑代码之间使用空行隔开，但不要滥用空行。

OOP 规约

  1. 直接用类名访问静态方法和静态变量。

  2. 重写方法必须加 @Override 注解。

  3. 尽量避免可变参数编程。

  4. 不使用过时的类或方法。

  5. 使用常量调用 equals() 方法，反之容易导致空指针异常。

  6. 包装类对象之间值的比较，采用 equals 方法。

  7. POJO 类属性必须使用包装数据类型。

  8. 所有局部变量使用基本数据类型。

  9. 定义 POJO 类时，不要设定任何属性默认值。

  10. 构造方法中禁止加入任何业务逻辑，可以放在 init 方法中。

  11. POJO 类必须写 toString 方法。

  12. 使用 String 的 split 方法得到数组时，要排查最后分隔符后有无内容：

  13. 多个构造方法按顺序放在一起。

  14. 类方法定义顺序：public / protected > private > getter/setter。

  15. 在 getter/setter 方法中，不要增加业务逻辑。

  16. final 使用场景：

    • 不允许被继承的类，如：String 类。

    • 不允许修改引用的域对象，如：POJO 类的域变量。

    • 不允许被重写的方法，如：POJO 类的 setter 方法。

    • 不允许运行过程中重新赋值的局部变量。

    • 避免上下文重复使用一个变量，使用 final 描述可以强制重新定义一个变量，方便更好 地进行重构。

  17. 慎用 Object 的 clone 方法拷贝对象：

    clone 方法是浅拷贝，对象内属性引用的对象只会拷贝引用地址，而不会将引用的对象重新分配内存，相对应的深拷贝则会连引用的对象也重新创建。

  18. 类成员和方法的访问控制：

    • 如果不允许外部直接通过 new 来创建对象，那么构造方法必须是 private。
    • 工具类不允许有 public 或 default 构造方法。
    • 类非 static 成员变量并且与子类共享，必须是 protected。
    • 类非 static 成员变量并且仅在本类使用，必须是 private。
    • 类 static 成员变量如果仅在本类使用，必须是 private。
    • 若是 static 成员变量，必须考虑是否为 final。
    • 类成员方法只供类内部调用，必须是 private。
    • 类成员方法只对继承类公开，那么限制为 protected。

集合处理

  1. hashCode 和 equals ：

    • 只要重写 equals，就必须重写 hashCode。

    • Set 存储不重复对象，依据 hashCode 和 equals 进行判断，所以 Set 存储对象必须重写这两个方法。

    • 如果自定义对象作为 Map 的键，那么必须重写 hashCode 和 equals。（String 类已经重写了 hashCode 和 equals 方法）

  2. ArrayList 的 subList 结果不可强转成 ArrayList。

  3. 使用集合的 toArray(T[] array) 方法，以将集合转成数组，不要使用 toArray() 无参方法。

  4. 使用 Arrays.asList() 把数组转换成集合时，不能使用其 add/remove/clear 方法以修改集合。

  5. 不要在 foreach 循环中进行元素的 remove/add 操作。remove 元素使用 Iterator 方式。

  6. 集合初始化时，要指定集合初始化大小。

  7. 使用 entrySet 遍历 Map 类集合，而不是 keySet 方式。

  8. 高度注意 Map 类集合 k/v 能不能存储 null 值

  9. 合理利用好集合的有序性(sort)和稳定性(order)，避免集合的无序性(unsort)和 不稳定性(unorder)带来的负面影响。

    ArrayList 是 order/unsort；HashMap 是 unorder/unsort；TreeSet 是 order/sort。

  10. 利用 Set 可以快速对集合进行去重。

并发处理

控制语句

  1. switch：

    在一个 switch 块中，每个 case 要么通过 break/return 来终止，要么通过注释标明程序执行到哪一个 case 语句为止。在一个 switch 块内，都必须放置一个 default 语句并放在最后。

  2. 在 if/else/for/while/do 中都必须使用大括号，即使只有一行代码。

  3. 尽量避免多层的 if-else 语句。

  4. 除方法参数外，不要再条件判断中执行复杂的语句，不易读。

  5. 循环语句要考量性能，定义变量等操作应当放在循环外。

注释规约

  1. 类、类属性、类方法的注释必须使用 Javadoc 规范（/** 内容 */）。
  2. 所有的抽象方法（包括接口中的方法）必须使用 Javadoc 注释，指出该方法具体做的功能。
  3. 所有的类都必须注释创建者和创建时间。
  4. 所有枚举类字段都必须要有注释，说明每个数据项的作用。
  5. 不要强行凹英文，重要在于表达清晰，用中文也挺好。
  6. 修改代码的同时，注释也需要进行相应的修改。
  7. 合理处理注释掉的代码，进行相应说明，若无用，再删去。
  8. 好的代码是是有很好的可读性的，是自解释的，注释仅仅知识辅助作用。

其他

  1. 在使用正则表达式时，利用好预编译功能。
  2. 使用 Math.random() 方法，要注意其取值范围是 [0, 1) 的浮点值，如果要获取整数，不要简单的乘以10来进行放大后再取整，而是使用 Random 对象的 nextInt 或者 nextLong 方法。
  3. 获取当前毫秒值使用 System.currentTimeMills() 方法。
  4. 不要再视图模板中加入任何复杂的逻辑。
  5. 及时清理不再使用的代码段或配置信息。
